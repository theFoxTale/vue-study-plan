# Module 0 · Теория: отличие `Vue` от `React` и `Angular`

Этот материал закрывает второй теоретический пункт `Module 0`: понять, **чем отличаются `Vue`, `React` и `Angular`**, почему они по-разному ощущаются в работе и в каких сценариях обычно выбирают каждый из них.

---

## 1. Коротко: что есть что

- `Vue` — **progressive framework** для интерфейсов с цельной официальной экосистемой.
- `React` — **UI-библиотека** с огромной экосистемой и высокой гибкостью.
- `Angular` — **полноценный framework/platform** со строгой архитектурой и большим количеством встроенных решений.

Главная разница не в том, какой инструмент "быстрее" или "лучше", а в том, **сколько решений он принимает за вас**.

---

## 2. Разница в философии

### `Vue`

`Vue` старается занять середину:

- ниже порог входа;
- понятный шаблонный синтаксис;
- официальный стек для основных задач;
- можно начать с малого и масштабироваться.

Идея `Vue`: дать разработчику **мягкий вход**, но не упереться в потолок, когда приложение становится серьезнее.

### `React`

`React` дает очень сильную и гибкую основу для UI, но часто оставляет выбор дополнительных решений на команде:

- чем роутить;
- как хранить состояние;
- как строить data layer;
- как организовать архитектуру проекта.

Это плюс для опытных команд, которым нужна свобода, но для новичка или небольшого продукта это может добавлять лишний шум.

### `Angular`

`Angular` берет противоположный курс:

- много вещей встроено сразу;
- есть строгие паттерны;
- выше объем концепций на входе;
- архитектура более формализована.

Это полезно там, где важны единообразие, стандартизация и длинная поддержка больших enterprise-проектов.

---

## 3. Сравнение по ключевым критериям

| Критерий                  | `Vue`                                              | `React`                                   | `Angular`                                        |
|---------------------------|----------------------------------------------------|-------------------------------------------|--------------------------------------------------|
| **Подход**                | Progressive framework                              | UI library + ecosystem                    | Full framework                                   |
| **Порог входа**           | Низкий/средний                                     | Средний                                   | Высокий                                          |
| **Синтаксис UI**          | templates + SFC                                    | JSX/TSX                                   | templates                                        |
| **Типизация**             | Отличная поддержка TS, но можно входить постепенно | Отлично сочетается с TS                   | TypeScript-first по духу                         |
| **Экосистема**            | Более цельная и официальная                        | Очень широкая, но более фрагментированная | Многое встроено                                  |
| **Архитектурная свобода** | Средняя                                            | Высокая                                   | Ниже, больше правил                              |
| **Скорость старта**       | Высокая                                            | Средняя                                   | Ниже                                             |
| **Когда особенно удобен** | Практичный продуктовый frontend                    | Гибкие продуктовые команды, широкий рынок | Enterprise и большие стандартизированные команды |

---

## 4. Как это ощущается в реальной работе

### Если вы берете `Vue`

Обычно вы быстрее доходите до момента "я уже делаю фичу":

- компонент;
- router;
- store;
- формы;
- API;
- потом `Nuxt`, если нужен SSR или SEO.

Во `Vue` чаще меньше споров на старте, потому что экосистема более согласована: `Vue Router`, `Pinia`, `Vite`, `Nuxt`.

### Если вы берете `React`

Обычно вы получаете:

- огромную экосистему;
- очень большой рынок вакансий;
- много готовых подходов;
- но и много архитектурных решений, которые нужно принимать самим.

`React` особенно хорош, когда команде важна максимальная гибкость и есть опыт выбирать инструменты осознанно.

### Если вы берете `Angular`

Обычно вы получаете:

- строгую структуру;
- мощный встроенный набор возможностей;
- понятные правила для больших команд;
- более тяжелый старт для человека, который только заходит во frontend-frameworks.

## 5. Как отличаются компонентные модели

### `Vue`

Во `Vue` компонент обычно выглядит как `Single File Component`:

```vue
<template>
  <button @click="count++">Clicked {{ count }}</button>
</template>

<script setup lang="ts">
import { ref } from 'vue'

const count = ref(0)
</script>
```

Особенности:

- шаблон похож на HTML;
- логика отделена от разметки, но живет рядом;
- `Composition API` удобно масштабируется.

### `React`

В `React` UI обычно пишется через `JSX`:

```tsx
import { useState } from 'react'

export function Counter() {
  const [count, setCount] = useState(0)

  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count}
    </button>
  )
}
```

Особенности:

- разметка пишется внутри JavaScript/TypeScript;
- высокая гибкость;
- для многих это очень естественно, но кому-то шаблонный стиль `Vue` воспринимается проще.

### `Angular`

В `Angular` компонент обычно сильнее завязан на framework conventions:

- decorator-based стиль;
- templates и class logic;
- DI, modules/standalone components, services и другие встроенные концепции.

Это делает систему более стандартизированной, но и заметно увеличивает когнитивную нагрузку на старте.

---

## 6. Router, state, data layer

Здесь различие особенно заметно.

### `Vue`

Чаще всего путь выглядит так:

- `Vue Router`
- `Pinia`
- `@tanstack/vue-query` при работе с server state
- `Nuxt`, если нужен meta-framework

Пакеты хорошо сочетаются и воспринимаются как естественное продолжение core stack.

### `React`

Чаще встречаются разные сочетания:

- `React Router`
- `Redux`, `Zustand`, `Jotai`, Context
- `TanStack Query`
- `Next.js`, `Remix` и другие meta-frameworks

Гибкость очень большая, но и решений больше.

### `Angular`

Многое идет как часть стандартного способа разработки:

- router;
- services;
- dependency injection;
- RxJS-подход;
- более framework-centric архитектура.

---

## 7. Когда что выбирать

### Когда разумно выбрать `Vue`

Выбирайте `Vue`, если:

- хочется быстро войти в современный frontend;
- нужен понятный и практичный стек;
- хочется меньше времени тратить на выбор инструментов;
- нужен путь от простых SPA до `Nuxt` и SSR;
- важен баланс между простотой и production-ready подходом.

### Когда разумно выбрать `React`

Выбирайте `React`, если:

- важна максимально широкая экосистема;
- команда любит конструировать стек под себя;
- нужен большой рынок библиотек и вакансий;
- комфортно жить в более "open-ended" среде.

### Когда разумно выбрать `Angular`

Выбирайте `Angular`, если:

- проект enterprise-ориентирован;
- важна строгая архитектурная дисциплина;
- команда готова принять более тяжелый вход ради стандартизации;
- в компании уже сильная `Angular`-экспертиза.

---

## 8. Почему в этом плане выбран именно `Vue`

Для учебного маршрута `Vue` удобен по нескольким причинам:

1. Он хорошо опирается на базовые `HTML`, `CSS`, `JavaScript`.
2. Позволяет довольно быстро увидеть результат.
3. Не перегружает новичка большим количеством обязательных решений в первый же день.
4. При этом не ограничивается "простыми проектами" и спокойно ведет к `TypeScript`, `Pinia`, тестам, архитектуре и `Nuxt`.

Именно поэтому для **спокойного, глубокого и системного** изучения frontend `Vue` часто оказывается очень удачным выбором.

---

## 9. Типичные ошибки в понимании

### "Vue только для маленьких проектов"

Неверно. У `Vue` давно зрелая production-экосистема, включая `Nuxt`, `Pinia`, `Vite`, тестирование и архитектурные практики.

### "React всегда лучше, потому что вакансий больше"

Количество вакансий важно, но это не делает инструмент автоматически лучшим для обучения. Иногда более цельная экосистема помогает быстрее дойти до реального уровня.

### "Angular устарел"

Тоже неверно. `Angular` остается сильным инструментом, особенно в enterprise-среде. Просто у него другой профиль задач и другой порог входа.

## 13. Мини-конспект

- `Vue` дает баланс между простотой старта и зрелостью экосистемы.
- `React` дает максимальную гибкость, но требует больше самостоятельных решений.
- `Angular` дает строгую архитектуру и встроенные решения, но заметно тяжелее на входе.
- Для этого плана `Vue` выбран как наиболее удобный путь к современному production-ready frontend без лишнего шума на старте.
